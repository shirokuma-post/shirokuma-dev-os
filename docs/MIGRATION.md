**削除したものはありません。** 移動と、発火条件の変更だけです。

| 1.5.1 | 2.0 での行き先 |
|---|---|
| `engineering-doctrine`（作者層 19,011字） | `docs/doctrine-full.md` — on-demand。自動発火しない |
| `engineering-doctrine-universal`（14,123字） | 核を `doctrine/core-2000.md` に蒸留（1,956字）。残りは `docs/` |
| `staff-officer`（19,794字） | `skills/staff-officer/` として存置。**常時ロードしない**。多 lane 時のみ |

## 何が変わるか

| | 1.5.1 | 2.0 |
|---|---|---|
| 毎回読ませる量 | 34KB+ | **約 2,000 字（+3.1%）** |
| 依頼の検査 | なし | **フック** |
| 成果物の検査 | なし | **フック（0 token）** |
| **完了の判定** | **AI の自己申告** | **検査が赤なら完了不可** |
| 決定の蓄積 | なし | **`DECISIONS.md` に機械的に回収** |

## なぜ 34KB を常時読ませるのをやめたか

同一条件で用量を振った実測（Haiku・圧力条件・各 6 run）:

| doctrine | 字数 | token 増 | 逸脱 |
|---|---|---|---|
| なし | 0 | — | 6/6 |
| 10 行 | 396 | +0.6% | 5/6 |
| **2,000 字版** | 1,956 | **+3.1%** | **0/6** |
| 34KB 版 | 33,917 | +59.4% | 2/6 |

**2,000 字が 34KB と同等以上を 1/19 のトークンで達成しました。**
効いていたのは思想でも業界用語でもなく、**実際に踏まれた逃げ道の明示列挙**でした。

2,000 字 → 34KB の追加 32KB（業界用語対応・思想史・n=1 の経験則）は、保護を改善していません。
**捨てていません。`docs/` にあります。** 実行時に毎回読ませる必要が無かった、というだけです。

## 移行手順

1. `BOUNDARIES.md` を書く（最重要。可否は書かなくてよい）
2. `.claude/invariants/` に、守りたい条件を実行可能な形で置く
3. `.claude/devos.json` に検査コマンドを書く（任意）
4. `node benchmarks/self/run.mjs` で自己検査が緑になることを確認
5. しばらく `SHIROKUMA_DEVOS=off` を手元に置いておく（誤検知が出たら即座に止められるように）
