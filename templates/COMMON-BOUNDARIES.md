# 境界を洗い出す

**このファイルの 7 問をそのまま使わないこと。** あれは 3 層白ラベル SaaS の例で、
個人向けアプリや社内ツールには当たらない。

**汎用なのは問いではなく、問いの作り方。** 手順は 3 段階。

---

## A. 総当たり表を作る（構造に依存しない生成器）

### A-1. 登場人物を挙げる

> **「このサービスには、どんな立場の人が出てきますか。」**
> お金を払う人 / 実際に使う人 / その人のお客さん / 社内で管理する人

1 種類しか出てこないなら、それでよい。境界はほとんど無い。

### A-2. 総当たりにする

挙がった立場を **すべての組み合わせ**で並べる。上下も横も区別しない。

```
立場が A / B / C なら
  A→B  A→C  B→A  B→C  C→A  C→B   ＋ B→B（同じ立場どうし）
```

### A-3. 各組に 2 問ずつ当てる

> **「〈X〉の人が、〈Y〉の◯◯を見られました。困りますか。」**
> **「〈X〉の人が、〈Y〉の◯◯を変えられました。困りますか。」**

◯◯ は**そのサービスで一番大事なもの**を入れる（売上 / 予約 / カルテ / 原稿 / 給与…）。

**これだけで、そのサービス固有の境界がひと通り出る。**
組が多すぎるなら、隣接する組と同列の組から先に聞く。

---

## B. 構造に依らない失敗の型（どんなサービスでも当てる）

総当たりで拾えない、**書き方の問題**から来るもの。4 つある。

### B-1. 自分のものだから、全部変えられる

> **「〈利用者〉が自分の登録内容を編集する画面で、
> 自分の役職や権限も一緒に変えられました。困りますか。」**

**唯一、階層が無くても必ず来る。** 会員機能があれば当たる。
「自分の行だから編集してよい」を通すと、変えてはいけない項目まで通る。

実例: 誰でも管理者に昇格できる状態だった。3 回直してようやく閉じた。

### B-2. 条件が、いつも成立してしまう

> **「まだ何にも紐づいていない人がログインしました。
> その人の画面に、他の人の情報が出ていました。困りますか。」**

絞り込みの条件を書き間違えると、**絞り込まれずに全部出る。**
「その行が持っている ID」を条件にすると、常に成立する。

実例: 担当 0 件の利用者が、他社の利用ログを 5 件読めていた。

### B-3. 横の線を忘れる

> **「同じ立場の〈A〉と〈B〉の間で、片方がもう片方のものを見られました。困りますか。」**

上下の線は意識するのに、**同列は抜ける。** A-2 の総当たりに必ず含めること。

### B-4. 一部の面だけ守られる

> **「本体のデータは守られていますが、
> **アップロードしたファイルの一覧 / 書き出した CSV / 通知メール / ログ画面**に
> 他の人のものが混ざっていました。困りますか。」**

線を引いても、**消費する面が全部乗るまで完了ではない。**
実例では、決めた 3 日後に 4 つの機能へ手作業で展開している。

---

## C. 例: 3 層白ラベル SaaS の場合

A と B を、運営 → 代理店 → 店舗 の構造に当てはめると 7 問になる。
**構造が違えば問いも変わる。これはそのまま使うものではなく、当てはめ方の見本。**

1. 〈スタッフ〉が自分の名前を変える画面で、自分の役職も変えられた（← B-1）
2. 〈代理店の担当者〉が契約している 10 店舗ぜんぶの売上と問い合わせを見られた（← A-3）
3. まだどの店舗の担当にもなっていない人に、他店の情報が出ていた（← B-2）
4. 〈A 店〉の店長が〈B 店〉の売上を見られた（← B-3）
5. お客さんに届いたメールの差出人が、店舗名でなく運営会社名だった（← A-3・白ラベル固有）
6. 逆に運営会社名が出ていないと困る場面はあるか（← 5 の例外）
7. アップロードした請求書の一覧に、他店のファイル名が混ざっていた（← B-4）

**5 と 6 は白ラベルがある場合のみ。** 無ければ聞かない。

---

## 聞き方のルール

**すべて「画面に何が映っているか」で聞く。** 仕組みの言葉は使わない。

| 使わない | 使う |
|---|---|
| テナント / 上位 / 下位 | 〈代理店の担当者〉〈お店の店長〉 |
| 所属 / 権限 / 認可 / RLS | どこにも登録されていない人 / 役職 |
| レコード / 列 | 自分の登録内容 |
| 可視性 | 一覧に出てくるか |

**答えは 3 つでよい。**「困る」「困らない」「**場合による / 分からない**」。
3 つ目も正しい答え。「この線の引き方は事業判断で決まる」と記録すれば、
**実装時に AI が空白を見つけて確認しに来る**（実測: 3/3 で停止）。

---

## 答え終わったら

1. `BOUNDARIES.md` に転記する
2. 決まった項目は `.claude/invariants/` に検出器を書く

### なぜ検出器まで作るのか

実運用 SaaS の 242 本の migration を分析した結果:

| | 収束するか |
|---|---|
| 線をどこに引くか | **する**。4 本で固まり、以後引き直していない |
| 新機能へその線を通すこと | **しない**。防御系の作業は 16% → 16% → 13% と比率が落ちていない |

隠れた分岐点は 2.5 ヶ月で 8 個。1 つあたり平均 2.6 箇所へ展開が必要だった。

**決めるのは人が 1 回。通すのは機械が毎回。**
