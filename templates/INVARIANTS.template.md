| ルール | 詳細 |
|---|---|
| **特権鍵は集約点の内側だけ** | {{例: service role key / 管理者 API キー}} は {{集約点（例: Gateway / BFF）}} 内のみ。他コンポーネントは持たない。CI ガードで再導入を禁止 |

## 4. デプロイ運用

| ルール | 詳細 |
|---|---|
| **正規デプロイ経路を1つに定める** | {{例: git push の auto-deploy}}。CLI は補助 |
| **デプロイ後は実測検証（assumption 禁止）** | {{例: commit status / 本番レスポンス}}で反映を確認。「push が通った＝OK」で済ませない（→ `engineering-doctrine` 規律3） |

| **ロールバック手順を即実行できる状態に** | {{例: 旧デプロイを Promote}} |

## 6. 依存管理（→ `engineering-doctrine` 規律6）

| ルール | 詳細 |
|---|---|
| **壊れると痛い依存は範囲指定を解除して完全固定** | {{例: 認証/暗号/決済/DBクライアント/AI SDK/フレームワーク本体}}。caret `^` / tilde `~` を外す |
| **複数サービスで同じ依存のバージョンを揃える** | 揃っていないと型保護もデプロイも環境依存の事故源 |
| **固定は放置ではない。定期に意図的アップグレード** | 例: 月次で `outdated` 確認 → 段階 upgrade → 型/ビルド検証 → 版整合レポート |

## 7. 責任境界（外部呼び出しは集約点経由）

> 系統が複数ある場合（例: 個人系 / 法人系）、各系統は**完全に分離した責任範囲**を持つ。系統を越境しない。

| ルール | 詳細 |
|---|---|
| **全ての外部呼び出しは自系統の集約点（Gateway）経由** | 各モジュールから外部 API・他モジュールへの直接 fetch は集約点 proxy 経由 |
| **他モジュールの本番 URL を直接 fetch しない** | hardcode した `https://other.example/...` は違反。proxy route に書き換え |
| **外部 API 直接 fetch 禁止** | {{例: Stripe/Meta/OpenAI/Anthropic}} は集約点の Vault / Secret Broker で復号 + 呼び出し |
| **系統越境禁止** | 系統 A が系統 B の集約点を叩く（またはその逆）は責任境界違反 |
| **例外: Webhook 受信 / 集約点同士の HTTP** | 受信は署名検証必須。集約点同士は HMAC 署名 + 認証境界明示 |

> ⚠️ 「便利だから」で他モジュールの URL を直接 fetch するのは責任境界の崩壊 = データ越境の温床。
> URL を各所で hardcode せず、集約点 URL を1個だけ env で持つ。
