# GOVERNANCE — 汎用開発OS の正典統治

> このプラグイン `shirokuma-dev-os` は、複数プロジェクトの **上流の正典**。
> 各プロジェクト（作者のプロダクト群 / 導入した外部プロジェクト）は、この正典を **継承し、固有差分だけを足す**。
> 二重管理（同じ普遍原理がプラグインとプロジェクトの両方に書かれ、やがてズレる）を防ぐための統治。
>
> **配布形態（v1.2 で確定）**: 本 repo は **2 層を同梱したまま配布する**。
> - **第 1 層（2.0 で降格）** = `docs/doctrine-full.md`。作者の n=1 経験則と思想史。**自動発火しない on-demand 資料**
> - **常時ロード層（2.0 で新設）** = `doctrine/core-2000.md`（1,956 字）。実測で 34KB と同等以上を +3.1% のトークンで
>   （技術スタック前提・実戦の具体例を含む）。外部読者には「規律がどんな失敗から生まれたか」を示す**実例集**として価値がある。
> - **第 2 層（発火条件を狭めた skill 群）** = `boundary-authoring` + `doc-constitution` + `staff-officer` + `session-operations` + `guardrail-authoring`。
>   固有名・スタック前提を剥いた普遍核。**外部プロジェクトで実際に発火させるのはこちら**。
>   （`guardrail-authoring` = 番人と品質ゲート。失敗カタログ + CI/番人雛形で「番人を緩めない」を実装として配給する層）
> 迷ったら: 読み物としては両層、運用に組み込むのは第 2 層 + templates。

## 1. 上下関係（継承構造）

```
上流（親・正典）= shirokuma-dev-os プラグイン   ← 普遍原理はここにしか書かない
        │  継承（子は上書きせず、固有差分だけ足す）
        ▼
下流（子）= 各プロジェクト（作者のプロダクト群 / あなたのプロジェクト） ← 固有設定だけ持つ。普遍部は親を参照
        │  還流（子で生まれた普遍教訓は親へ昇格）
        ▲────────────────────────────────────────┘
```

- **上流（親）= このプラグイン**。普遍原理（思考様式・文書憲法・参謀フロー・不変ルールの普遍部・テンプレ）は **ここにしか書かない**。
- **下流（子）= 各プロジェクト**。固有設定だけを持つ。普遍部は親を **参照** し、コピーしない。
- 子は親を **上書きしない**。違う運用が要るなら「固有差分」として足す（親は変えない）。

## 2. 正典マップ（どの事実がどちらにあるか）

| 事実 | 正典の所在 |

| **完了主張の統制（Claim Integrity）・四値判定** | **親**: `hooks/stop.mjs`（実装が正典。主張は `benchmarks/` で再現可能） |
| **依頼の受け口検査・境界注入** | **親**: `hooks/intake.mjs` |
| **Intent Anchor（意図の先出し宣言）** | **親**: `hooks/intake.mjs` の境界分類。**2.0 で内省ベースから境界分類ベースへ変更**（実測: 昇格 0/3 → 3/3） |
| 文書運用ルール（7条 + Tier 読み分け） | **親**: `skills/doc-constitution` |
| 参謀フロー（5層 × 4ライン・検品・ゲート・振り返り） | **親**: `skills/staff-officer`（振り返り 5 軸の雛形 = `templates/RETROSPECTIVE.template.md`） |
| マルチセッション運用（1ドメイン1セッション/起動/撤退/handoff） | **親**: `skills/session-operations` |
| 番人と品質ゲート（失敗カタログ + CI/番人雛形） | **親**: `skills/guardrail-authoring` |
| 不変ルールの普遍部（セキュリティ核/性能核/…） | **親**: `templates/INVARIANTS.template.md` |
| プロジェクトの北極星・指揮系統の骨 | **親**: `templates/CLAUDE.template.md` |
| 製品名・project ID・URL・team・鍵名 | **子**: 各プロジェクトの `CLAUDE.md` / `INVARIANTS.md` |

> 落として「骨（なぜそうするか＝原理）」だけ親に残す。

## 3. 差分ルール（子の書き方）

- 子は普遍部を **コピーしない**。親を参照する（例: 「思考様式は `shirokuma-dev-os` のスキル群に従う」の1行）。
- 子に書くのは「親と違うこと」「親に無い固有のこと」だけ（→ `doc-constitution` 3条）。
- 子の `CLAUDE.md` 冒頭に **継承宣言** を置く: 「普遍原理の正典は `shirokuma-dev-os`。本ファイルは固有差分」。

---

## 4. 還流ルール（子 → 親への昇格）

プロジェクトで新しい教訓・ルールが生まれたら、それが **普遍か固有か** を判定する:

1. **固有名を剥けるか** — 製品名・ID を消しても意味が残るなら普遍候補。残らないなら固有（子に置く）。
2. **統合チェック**（→ `doc-constitution` 5条）— 既存の規律・条と束ねられないか先に見る。新規追加は最後の手段。
3. **昇格** — 普遍なら親（該当スキル / テンプレ）に規律・条として追記する。
4. **重複除去** — 昇格したら、子側の元記述は親への **参照に置き換える**（実体を2箇所に残さない = 1事実=1正典）。

> 還流の実例（2026-05-31）: 「仕組みを作った≠完了」は元々 `staff-officer` だけにあった。
> これを思考の構えとして `docs/doctrine-full.md` 規律5 に昇格し、`staff-officer` 側は
> 「正典は規律5」と参照しつつ参謀実務の差分だけに置き換えた。これが還流の型。

### 還流は月次ルーティン（不定期にしない）

還流を「気づいた時にやる」運用にすると滞留する（実例: 2026-06-30〜07-07 の教訓群が未還流のまま 1 週間超滞留し、棚卸しが必要になった）。**月 1 回**、以下を回す:

1. 下流プロジェクトの教訓ノート（feedback 系 memory）を前回還流以降の分だけ走査
2. 各件に **固有名剥がしテスト**（§4-1）→ 普遍候補を抽出
3. **統合チェック**（§4-2）→ 既存規律への追記 or 新設を裁定し、親へ昇格 + 子を参照化
4. 本 GOVERNANCE の「最終確認日」を更新し、CHANGELOG に還流内容を記録（version bump）

## 5. ズレ検知（乖離を防ぐ運用）

- 親（プラグイン）を更新したら「子に同じ普遍記述のコピーが残っていないか」を確認する。
- 子で「あれ、これ親と違う」と気づいたら、それは ①親が古い → 親を直す ②子が固有 → 差分として明示、のどちらか。
  放置 = 二重管理の再発。
- 定期点検（例: 週次/月次の監査）で「子に普遍記述のベタ書きが復活していないか」を見る。
- **親の更新は全プロジェクトに即時反映される**（プラグインはグローバル）。だから親を直す時は、
  既存プロジェクトの固有差分と矛盾しないかを意識する（普遍部だけ触る・固有判断は子に委ねる）。

---

## 6. このプラグインの育て方

- 普遍原理が増えたら親に昇格（4節）。**固有名は絶対に親へ持ち込まない**（他プロジェクトで誤作動する）。
- スキル/テンプレを更新したら、本 GOVERNANCE の正典マップ（2節）も同じ変更で直す（→ `doc-constitution` 2条 鮮度＞完全性）。
- 親は git 管理 + GitHub 公開 repo。更新は version bump（`.claude-plugin/plugin.json`）+ `CHANGELOG.md` 記録 + `node scripts/audit-self-integrity.mjs` PASS を 1 セットで行う。
