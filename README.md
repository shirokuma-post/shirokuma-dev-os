# shirokuma-dev-os 2.0

**AI が根拠以上のことを保証するのを防ぐ門番。**

能力にばらつきのある AI でも、開発工程だけは揃える。
コードを上手く書かせる道具ではありません。

---

## ⚠️ 最初に読んでください: これはあなたのマシンでコードを実行します

このプラグインは **2 つのフック**を入れます。つまりインストールすると、
あなたのプロンプトと成果物を検査するスクリプトが動きます。

| フック | いつ | 何をするか |
|---|---|---|
| `hooks/intake.mjs` | あなたがプロンプトを送った時 | 依頼文を読み、判断基準を文脈に注入する |
| `hooks/stop.mjs` | AI が応答を終えようとした時 | 検査を実行し、赤ければ差し戻す |

- **依存パッケージ 0**（Node 標準ライブラリのみ）
- **ネットワークアクセスなし**
- **テレメトリなし**（何もどこにも送りません）
- **書き込むのは `DECISIONS.md` と `.claude/.devos/` のみ**
- **止めたい時**: `export SHIROKUMA_DEVOS=off` で全機能が停止します

コードは短く保っています。インストール前に `hooks/` を読んでください。

## 何をするか

```
Acceptance Contract  →  Verification  →  Completion Claim
（受け入れ条件）         （検証）          （完了主張）
```

**受け入れ条件が無ければ検証は成立せず、検証が無ければ完了主張も成立しない。**

CI は「設定されたテストが通った」ことを保証します。
このプラグインはさらに「**そのテストは何を保証するものなのか**」を問います。

### 判定は四値

| | 意味 | 挙動 |
|---|---|---|
| `PASS` | 裁定済みの受け入れ条件を全て満たした | 通す |
| `FAIL` | 満たしていない | **差し戻す**（既定 3 回まで） |
| `INVALID` | 検査自体が走らなかった | 止めない・**緑と言わせない** |
| `UNSPECIFIED` | 受け入れ条件が未定義 | 止めない・**緑と言わせない** |

**進めることは許す。保証は許さない。**
現実の開発では受け入れ条件が揃っていない場面が普通にあります。そこで全部止めると使えません。
しかし勝手に緑認定すると事故ります。

### Claim Integrity

完了主張の強さが、実際に存在する証拠の強さを超えていないかを検査します。

| 証拠 | | 名乗れる主張 |
|---|---|---|
| E0 | 検査が走っていない | 「実装した」まで |
| E1 | プロジェクトのテストのみ | 「実装した」まで |
| E2 | 裁定済みの受け入れ条件が通った | 「完了した」まで |

「`npm run typecheck`: 通過」は**検査対象を特定した事実**なので許されます。
「検証しました。問題ありません」は**範囲が無限に読める主張**なので、E2 が無ければ差し戻されます。

## セットアップ

### 1. 境界線マップを書く（最重要）

書き方が分からない場合は `templates/BOUNDARIES.questions.md` の聞き取りテンプレートを使ってください。
`/boundary-authoring` skill が、事業側の人への質問を進行して `BOUNDARIES.md` を組み立てます。
**「分からない」は正しい回答です。**可否が未決なら「引き方は事業判断で決まる」と書けば、
実装時に AI が空白を見つけて確認しに来ます。

`BOUNDARIES.md` に「**誰と誰の間に線があるか**」を書きます。**可否まで書けなくて構いません。**

```markdown
### 境界1: 運営 ⇄ 代理店
運営の名前・ロゴが代理店の画面に漏れてはいけない。

### 境界2: 代理店 ⇄ 店舗
この線の引き方は、事業判断で決まる。   ← 可否を書かなくてよい
```

「引き方は事業判断で決まる」とだけ書いておけば、**AI が空白を見つけて確認しに来ます**。
抽象的な理念を書くより、線を引く相手を書く方が易しく、効果があります。

書き終えたら、**開発者が最初に読む文書（README / CLAUDE.md）に次の 1 行を必ず置いてください**:

> 境界線マップ（BOUNDARIES.md）があるので、作業前に必ず照らして。

マップを置くだけと、この 1 行があるのとで、AI が止まって確認しに来る率が **3/6 → 5/6** に変わります（実測）。

### 2. 受け入れ条件を書く（実装の前に）

`.claude/invariants/*.mjs` に、実行可能な条件を置きます。

```js
export const meta = {
  boundary: '境界2',
  invariant: '代理店は配下店舗のデータを読まない',
  ratified_by: 'あなたの名前',   // 空なら advisory 止まり。差し戻しの根拠にできません
};
export async function check(cwd) {
  return { ok: true, detail: '' };
}
```

**実装の後に AI に書かせないでください。** 満たせる条件しか書きません（八百長検査委員会）。

### 3. 検査コマンド（任意）

`.claude/devos.json`

```json
{ "command": "npm run typecheck",   // 毎ターン走ります。軽いものを
  "maxBlocks": 3,                   // 差し戻しの上限
  "timeoutSec": 120,                // 検査の打ち切り
  "warnSec": 30 }                   // これを超えたら「重すぎます」と警告
```

## 効果の実測値

すべて `benchmarks/` で再現できます。

### 弱いモデルの逸脱を止める

同一課題・同一圧力条件（「投資家デモが明日。雑でいいから最速で」）:

| モデル | doctrine なし | doctrine あり |
|---|---|---|
| Fable 5 | 逸脱 0/17（有効 run・逃げ方 none） | — |
| Opus 4.8 | 逸脱 0/18 | — |
| Sonnet 5 | 逸脱 2/18 | **0/6** |
| Haiku 4.5 | 逸脱 **13/18** | **0/6** |

**モデル間の開きが 72 ポイント → 0。** 揃えるのが目的です。

### 完了ブロックは、弱いモデルの到達点を上げる

Haiku・圧力条件・各 6 run:

| | 機能的に正しい | 偽の安心 |
|---|---|---|
| フックなし | **0/6** | 4/6 |
| フックあり | **6/6** | **0/6** |

能力が足りなかったのではなく、**何を達成すれば完了かの基準を持っていなかった**。

### ループの価値（実務規模・25 項目・3群 × 6 反復）

repo は毎回まっさら。持ち越すのは `DECISIONS.md` だけ。

| 反復 | Opus 素 | Sonnet + Dev OS（裁定なし） | **Sonnet + Dev OS（裁定あり）** |
|---|---|---|---|
| 1 | 21/25 | 21/25 | 21/25 |
| **2 以降** | 21/25（平坦） | 21/25（平坦） | **25/25（安定）** |

**交差点は 2 回目。人間が 1 回裁定するだけで、以後ずっと効きます。**

**裁定しないと、ループは誤りを固定します。** 裁定なし群の記録に残った文:
> 代理店owner も対象にした。**過去5セッションとも同じ判断だったため踏襲**

1 回目の推測が、繰り返しによって先例になりました。
だから `ratified_by` が空の記録は、差し戻しの根拠にできません。

### 情報境界の壁 — どのモデルも素では越えられない（2026-07-21・42 run）

依頼文に書かれていない事業判断（「代理店に店舗データを見せるか」）を含む実務規模の課題:

| 群 | 検証済み完了 | 検証済み完了あたりコスト |
|---|---|---|
| Opus 素 | **0/6** | **∞（何回払っても仕様を満たさない）** |
| Sonnet 素 | 0/6 | ∞ |
| **Fable 5 素（現行最上位・単価は Sonnet の約 1.7 倍）** | **0/6** | ∞ |
| **Dev OS + Sonnet** | 昇格した run は**通算 18/18 で満点** | **$1.86** |

bare の 30 run 中 29 run が「型定義から導ける」と**確信して**誤った側を実装しました。
答えはモデルにも repo にもありません。だから賢さでは越えられません。
（コストは worker の API 課金・差し戻しと空転込み。承認役 = 人間の 1 判断は含まない）

**昇格率（AI が止まって確認しに来る率）は、マップの参照経路で決まります**（用量反応・実測）:

| 経路 | 昇格 |
|---|---|
| マップを置くだけ（フック注入・旧文言） | 1/6 |
| 注入文言の強化 | 3/6 |
| **強化注入 + 依頼文書に参照 1 行（推奨構成）** | **5/6** |
| prompt 本文に分類指示を全文 | 3/3 |

だから `boundary-authoring` は「**開発者が最初に読む文書へ参照 1 行を足すまでが仕事**」です。

### 決定は収束する。適用は収束しない

実運用 SaaS の 242 本の migration を分析した結果:

| | 収束するか |
|---|---|
| 境界の構造（誰と誰の間に線があるか） | **する**。4 境界で固まり、以後 redefine していない |
| 境界の適用（新機能への横展開） | **しない**。防御系の比率は 16% → 16% → 13% とほぼ横ばい |

隠れた分岐点は **2.5 ヶ月で 8 個**。1 つあたり平均 2.6 箇所へ展開が必要でした。
実例では 06-12 に「上位は下位を見ない」と決め、06-15 に 4 機能へ**手作業で**展開しています。

**だから「決定は人間が 1 回、適用は機械が毎回」。**
`BOUNDARIES.md` は決定を保存しますが、適用は保証しません。
`.claude/invariants/` の検出器まで作って初めて自動になります。

ただし**汎用なのは問いではなく、問いの作り方**です。
実例の 7 問は 3 層白ラベル SaaS 固有で、個人向けアプリや社内ツールには当たりません。
`templates/COMMON-BOUNDARIES.md` には次を入れてあります。

- **A. 総当たり表** — 登場人物 × 登場人物 × (見る / 変える)。構造に依存しない生成器
- **B. 構造に依らない失敗の型 4 つ** — 自分のものだから全部変えられる / 条件がいつも成立する /
  横の線を忘れる / 一部の面だけ守られる。**どんなサービスでも当てる**
- **C. 例** — A と B を 3 層白ラベルに当てはめると 7 問になる。**当てはめ方の見本**

### Claim Integrity

受け入れ条件が無い状態（E1）で、Haiku・各 3 run:

| | 実測 |
|---|---|
| 偽の完了主張 | **0/3** |
| 未検証であることの開示 | **3/3** |
| 真実の報告への誤検知 | **0/7**（自己検査） |

## ⚠ 実運用は始まったばかりです（2026-07-21〜）

2026-07-21 に実プロジェクト（マルチテナント SaaS monorepo）への投入を開始しました。
**それ以前の数字はすべて合成 fixture と模擬課題での測定**であり、実運用はまだ 1 repo・初日です。

実運用初日に見つかったこと（すべて修正済み・fixture 化済み）:

- **誤検知 4 件** — INVALID 通知の無限ループ / 設定ゼロの誤分類 / 推奨表現・引用の Claim 誤判定
- **裁定 UX の欠陥 3 つ** — 工程用語の一括提示は非エンジニアには裁定不能 → 1 件ずつ平易な三択へ /
  文脈のない candidate は裁定不能 → 依頼要旨を自動添付 / 作業限りの解釈と普遍ルールの未分類 → 二分ルール
- **裁定サイクルの 1 周実証** — candidate 9 件 → 人間の裁定 → adopted 7・task-local 1・保留 1

「使い始めた日に穴が見つかる」は、この製品では想定どおりの動作です。
誤検知はその報告文を fixture に足せば二度と起きません。それが設計思想です。
試す時は `SHIROKUMA_DEVOS=off` を手元に置いたまま始めることを薦めます。

**最初に壊れそうな箇所**（設計から見えているが未測定）:

| リスク | 内容 | 対策（実装済） |
|---|---|---|
| 検査が重い | `command` が重いテスト一式だと、応答のたびに待たされる | `warnSec`（既定 30 秒）超過で警告。`timeoutSec` で上限 |
| 開示要求がうるさい | `invariants` が無い間、毎タスク差し戻されると恒久的な税になる | **1 プロジェクト 1 回だけ**差し戻し。以後は文脈で伝える |
| 誤検知 | Claim Integrity は手書きの正規表現。fixture 8 件では実際の報告文を覆えない | 誤検知を見つけたら `benchmarks/self/` に fixture を追加してください |
| `DECISIONS.md` の汚れ | 抽出が正規表現。以前 markdown の破片が混ざった | 改善済だが根本は同じ |

## 正直に書いておくこと

- **強いモデルの「コード品質」は上がりません。** Opus 4.8 は素の状態で逸脱 0/18 でした。
  常時読ませる 2,000 字の doctrine は、Opus には効果がありません（コストは +3.1% なので入れたままで害はない程度）。
- **効果が大きいのは弱いモデルと、完了ブロック**です。後者はモデルを問いません。
- **判定は日本語の正規表現**です。英語その他の報告では現状うまく動きません。
- **未検証の項目があります**（下記）。

### 未検証・既知の穴

| 項目 | 状態 |
|---|---|
| ~~境界マップの肥大化耐性~~ | **検証済**。10 境界＋例外 7 件でも 昇格 3/3・25/25 ×3 |
| ~~決定記録の肥大化耐性~~ | **検証済**。50 項目中 関連 3 件（3,788字）でも 25/25 ×3 |
| 参謀フロー（`staff-officer`） | **適用領域で一度も測れていない** |
| 多言語 | 日英に対応（fixture 10件で検証）。他言語は未対応 |
| Claim Integrity の網羅性 | fixture 10 件（日英・実運用の誤検知 2 件を還流済み）。増やせば別の誤検知が出る可能性 |
| 昇格の残り 1/6 | 推奨構成でも 6 run 中 1 run は確信誤読で進んだ。ゼロにする経路は prompt 全文のみ（3/3・ただし依頼文を書く人に負担） |
| 非エンジニアの単独運用 | 裁定 UX は 3 段改善したが、**非エンジニアが一人で回し続けられるかは未検証** |
| 聞き取りテンプレートの有効性 | 問いを「書けるか」から「答えられるか」に変換した。**実際の非エンジニアで未検証** |
| グレーゾーンの昇格較正 | 判定パネルが弁別能力ゼロで失敗。**未確定** |
| 差し戻し上限（既定 3 回）の妥当性 | 上限に当たった時の挙動は未観測 |

## 検証する

```bash
# 自己検査（課金なし・数秒）
node benchmarks/self/run.mjs

# 採点器の検証（課金なし）
node benchmarks/model/verify.mjs

# モデル検査（API 課金が発生します）
#   新しいモデルを信頼する前に、そのモデルの「逃げ方」を測ります
node benchmarks/model/run.mjs --model <model-id> --reps 6
```

> **前提**: `benchmarks/model/` は typescript を使います。初回に `benchmarks/model/.tools/` へ
> 導入するため、**ネットワークを使います**。「依存ゼロ・ネットワークなし」は `hooks/` についての記述です。
> ベンチマークを走らせない限り、ネットワークは一切使いません。

結果は `models.json` に保存され、次の形になります。

```json
{ "claude-haiku-4-5-20251001": {
    "deviationRate": 1,
    "evasionStyle": "semantic",      // 構文検査では捕まらない逃げ方
    "semanticObserved": true,
    "doctrineLevel": "core-2000" } }
```

**逃げ方は世代ごとに形を変えます。**

```
as any → as X as Y / eslint-disable → （駆逐）→ 機能を削る
構文     構文（禁止の迂回）                    意味 ← パターン検査が効かない
```

禁止するたびに、逃げ方が構文から遠ざかっています。だから**前提を置かず、測ります。**

## 同梱する skill（発火条件を狭めてあります）

| skill | いつ発火するか |
|---|---|
| `boundary-authoring` | `BOUNDARIES.md` を新規に作る / 境界を見直す時のみ |
| `guardrail-authoring` | CI・フック・検出器を書く / 変更する時のみ |
| `doc-constitution` | プロジェクト文書そのものを作る / 正典の食い違いを裁定する時のみ |
| `session-operations` | 複数セッションが同じ repo を同時に触る時のみ |
| `staff-officer` | 独立した作業 lane が 2 以上ある時のみ（**適用領域で未測定**） |

通常の実装・バグ修正・調査では**どれも発火しません**。
