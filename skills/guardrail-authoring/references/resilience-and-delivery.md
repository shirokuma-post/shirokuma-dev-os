# 回復力 & 安全なリリース — データ事故と壊れたデプロイを不可逆にしない

> Tier: T1 ・ 1行責務: バックアップ復元演習・破壊的でない移行・feature flag/即ロールバック・DAST・SBOM を足し、「不可逆な失敗」を可逆にする
> 位置づけ: guardrails の層3(デプロイ前)と層4(実行時)の補強。前提の優先順位 = **不可逆 ＞ 可逆**。データ消失・復元不能・止血不能なリリースは、どんな機能より先に潰す（顧客の安全 ＞ 効率）。

## A. データの回復力（最優先・有料顧客がいるなら必須）

**A1. テストしていないバックアップは「バックアップではない」**
- 失敗: 自動バックアップが有効でも、いざ復元したら壊れている/手順を誰も知らない/RTO が想定外に長い。
- 恒久: **復元演習(restore drill)を定期実施**。別環境へ実際に復元し、データ整合と所要時間(RTO)を計測。PITR(point-in-time recovery)が使えるか実機確認。
- 番人: ◯ 四半期ごとの復元演習をカレンダー化（`assets/packs/supabase-postgres/restore-drill.md.template`）。「最後に復元成功した日」を記録。

**A2. 破壊的な1段移行（expand-contract を使わない）**
- 失敗: `DROP COLUMN`/`RENAME`/型変更を1回の migration＋デプロイで実行 → 旧コードが動いている瞬間に壊れる/ロールバック不能。
- 恒久: **expand → migrate → contract** の3段。①新カラム追加(expand・後方互換)②両書き込み＋バックフィル③旧参照が消えてから削除(contract)。各段は単独でロールバック可能に。破壊的 DDL の前に必ずバックアップ。
- 番人: ◯ migration レビューで「この1段は旧コードと共存できるか」を必須チェック。`DROP`/`RENAME` を含む migration は要レビューフラグ。

**A3. 危険な操作の前にスナップショット**
- 恒久: 本番への破壊的 migration・一括 UPDATE/DELETE の直前にスナップショット取得を手順化。`DELETE`/`UPDATE` は必ず `WHERE` とトランザクション、まず `SELECT` で件数確認。

## B. 安全なリリース（高速なエージェント運用ほど効く）

> AIエージェント艦隊で速く出すほど、人のレビューがすり抜ける。だから「ヤバければ即止める」手段が**速度の前提**になる。重い canary 基盤より、まず軽量版を。

**B1. feature flag（暗闇で出して、スイッチで殺す）**
- 恒久: 新機能は flag の裏に出す（ship dark）。問題が出たら**デプロイし直さず flag を切る**＝秒で止血。flag は kill switch も兼ねる。
- 軽量実装: env / DB 1テーブル / 設定ファイルの boolean で十分（専用 SaaS は後で）。flag の評価は1箇所に集約。
- 番人: ◯ 恒久化した flag は定期棚卸し（消し忘れ＝負債）。

**B2. 即ロールバックを「手順書の1コマンド」にする**
- 失敗: 障害時に「どう戻すんだっけ」を調べている間に被害が拡大。
- 恒久: ロールバック手順を**事前に1コマンド化**して runbook に固定（alias 切り戻し / 直前デプロイへ redeploy）。ロールバックの**判断トリガー**（例: エラー率SLO侵食・5xx急増）も事前に決める。
- 番人: ◯ デプロイ runbook に「戻し方」と「戻す条件」を必須記載。

**B3. preview/staging を本番の前に必ず通す**
- 恒久: 本番前に preview 環境で E2E コアフロー＋DAST を通す。preview の URL を実叩きしてから昇格。
- （成熟段階）canary デプロイ＝一部トラフィックに先行配信して指標を見る。基盤が要るので Later。

## C. DAST（動いているアプリを外から叩く・SASTの穴を埋める）
- SAST は静的、DAST は**起動中のアプリ**に攻撃リクエストを撃つ。公開LP/フォーム/認証フローの実挙動の穴（ヘッダ欠落・反射型XSS・open redirect）を拾う。
- 軽量: **ZAP baseline scan** を preview URL に対して CI で（受動スキャン中心で誤検出少なめ）。`assets/packs/universal/dast.yml.template`。
- （成熟段階）認証付きフルスキャン・ファジング（入力の異常系大量投入）は Later。

## D. SBOM & 供給網の来歴（インシデント時に遡れる）
- **SBOM**: `syft` 等でビルド成果物の依存一覧を生成・保存。CVE が後で公表された時「どのリリースに入っていたか」を即特定できる（1コマンド・低コスト＝Now でよい）。
- （成熟段階）provenance(SLSA)・artifact 署名(cosign) は配布や調達要件が出てから。

## E. 秘密のローテーション（周期を決める＝「いつか」を消す）
- 恒久: 鍵・トークンのローテ周期を数値で決める（例: 署名鍵90日 / 漏洩時即時 / 退職・委託終了時即時）。手順を runbook 化（無停止ローテは NEXT/CURRENT 2鍵運用）。
- 番人: ◯ 「最終ローテ日」を台帳化し、周期超過を棚卸しで検出。
