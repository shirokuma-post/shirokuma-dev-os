# ローンチ前チェックリスト — テンプレート

> Tier: T1 ・ 最終確認日: YYYY-MM-DD ・ 1行責務: 課金/公開を始める前に機械的に通す出荷ゲート
> 使い方: ローンチ直前に**機械的に通すゲート**として使う。総合評価（良し悪しの議論）とは**分離**し、各項目を「無ければローンチしてはいけない」ハードゲートにする。固有値は読み替え。

> 原則: 機能・堅牢性・安全性が水準でも、**商取引の外皮**（課金結線・法令表記・申込導線）が欠けると課金開始が法令リスクになる。総合評価と実行チェックを混ぜると抜ける。

## 1. 商取引の外皮（法令・課金）
- [ ] 課金 UI が結線され、料金→決済→契約状態反映が**実フロー**で通る
- [ ] 申込口（CTA）が存在し導線が繋がっている
- [ ] 法令表記（特商法表記/利用規約/プライバシーポリシー等）が公開され、申込導線から到達可能
- [ ] 旧課金・旧機能の残骸が新しい境界に漏れていない

## 2. 番人が緑
- [ ] 秘密スキャン（pre-commit ＋ CI）緑、リポに秘密/PII なし
- [ ] 禁止パターン番人 0 件（最高権限キーの越境 import / 他コンポーネント直 fetch / `.env.example` の別系統 default）
- [ ] 型エラー 0 ・ lint error 0 ・ `any` baseline 増加なし
- [ ] 本番ビルド job が緑（tsc 緑だけを根拠にしない）
- [ ] 認証バイパス番人 緑（署名検証の短絡・body userId 信頼なし）
- [ ] テナント分離番人 緑（RLS 無効テーブル 0・テナント列違反 SQL 0 行）
- [ ] 実行時間上限番人 緑（プラン上限超過 route 0 件 ／ または有料昇格済）
- [ ] 依存バージョン突合 緑（コンポーネント間ドリフトなし）

## 3. 動作の実測
- [ ] E2E コアフローが**本番直叩き**で緑（越境0・データ汚染0）
- [ ] 全コンポーネントの `/health` が契約準拠（top-level `status` 3値）
- [ ] デプロイ実測検証: ビルドステータス success **かつ** 本番 health の version=HEAD 一致
- [ ] 死活監視 cron が全コンポーネントで常駐、異常時に管理者通知が届く（自系統のみ・dedupe）

## 4. 運用準備
- [ ] テスター案内（試すフロー/既知の暫定回避策/報告様式）が整備済
- [ ] ロールバック手順（alias 切り戻し/redeploy）が手順書にある
- [ ] テストアカウントに機密を入れない・本番 SNS 等への流出注意が周知済

---
> 1つでも未達ならローンチしない。チェックは「総合的に良いか」ではなく「これが無いと出してはいけない最低線」を測る。
