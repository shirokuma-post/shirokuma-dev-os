# session-operations — マルチセッション運用の型

> AI セッションを複数並走させる時の失敗は 2 種類ある。
> **解釈ズレ**（同じドキュメントを読んでも別セッションは微妙に違う初期解釈を出す）と、
> **共有状態の衝突**（同 repo / 同 DB / 同 live への同時操作で混戦）。
> どちらも「分離すれば安全」という直感の逆で、**分離そのものが生む**失敗モード。
>
> このスキルは [[staff-officer]]（1 セッション内の指揮系統）の外側、**セッション間**の統治を担う。

---

## 思想 (= 一本の幹)

**「分離の安全性を上げる」より「1 セッションで通せる距離を伸ばす」方向で設計する。**

セッション分離は文脈肥大による劣化を回避するが、解釈ズレと共有状態の衝突という別の失敗モードを持ち込む。分割は最後の手段。分割するなら、下の 5 本柱で境界を統治する。

---

## 5 本柱

### 1. 1 ドメイン = 1 セッション

- **ドメイン = 共有状態の最小単位**（同 repo / 同 DB schema / 同配布ライブラリ / 同 live URL）。
- 同じドメインを 2 セッションで同時に触らない。特に「複数箇所へ配布される正典ファイル」と「集約点 (Gateway 等) の横断変更」は厳守。
- 別ドメインなら並走してよい（例: 別 repo のフロント改修と別 repo のバッチ改修）。

### 2. 起動プロトコル（30 秒）— 事後訂正は全やり直し・起動時訂正は無傷

新セッション開始時に必ず 3 step:

1. **live 実測**（推測禁止）: 対象システムの health/version 実測 + `git log -5 --oneline` + `git worktree list` / `git branch --show-current` で他セッションの作業痕跡を確認
2. **記録読み込み**: 直近の引き継ぎノート（後述）を最優先で読む → プロジェクト memory の該当エントリ
3. **解釈確認（1 行提示）**: 「今の理解は◯◯と認識してます」を依頼者に 1 行で提示 → 訂正をここで吸収してから着手

### 3. 撤退シグナル（出たら分離をやめて統合セッションへ昇格）

- 依頼者からの訂正が **2 往復以上** 発生
- live 実測で **別セッションの作業痕跡** を発見
- 記録に「別セッションで決めた」という記述が **複数** 見つかる
- **同じファイルへの編集衝突**

1 つでも出たら、その作業は分離に向いていない。統合して 1 セッションで通す。

### 4. 引き継ぎノート（handoff）

セッションを跨ぐなら `session_handoff_<topic>_<date>.md` を必ず書く。必須 4 項目:

- **決定事項**（なぜそうしたか含む）
- **残課題**（次セッションがやること）
- **次が読むべき 1 ファイル**（読み筋を 1 本に絞る）
- **live の現状**（version / 深刻な未解決事象）

### 5. 共有 index の書込直列化

- 常時ロードされる index 文書（memory index 等）への書込は **主セッションのみ**。衛星セッションは read-only。
- 並列書込は「どちらの記述が正か」を破壊する（→ `doc-constitution` 1 条の並走版）。

---

## 使い方（チェックリスト）

```
□ このタスクは既存セッションのドメインと重なっていないか？（重なるなら分離しない）
□ 起動 3 step（live 実測 → 記録 → 解釈 1 行提示）をやったか？
□ 撤退シグナルが出ていないか？（出たら統合へ昇格）
□ セッションを跨ぐ前に handoff（4 項目）を書いたか？
□ 共有 index を衛星セッションから書いていないか？
```
