# doc-constitution — ドキュメント運用の憲法

> ドキュメントは「書いて終わり」ではなく**運用する資産**。
> 正典が乱立し、古い記述が放置され、同じ事実が複数ファイルにコピペされると、
> ドキュメントは信頼を失い「読まれない・嘘をつく」状態になる。それを防ぐ7条。
>
> **新規プロジェクト立ち上げ時**は、このプラグイン（`shirokuma-dev-os`）同梱の `templates/` から
> 3点（`CLAUDE.template.md` / `INVARIANTS.template.md` / `DOC_CONSTITUTION.template.md`）をコピーし、
> 各ファイルの `{{}}` に固有値を埋める。
> **6条の本文の正典はこのスキル**。`DOC_CONSTITUTION.template.md` は本文をコピーせず、
> プロジェクト固有の Tier 割り当てと正典マップだけを持つ（= 1条「1事実=1正典」・3条「子は差分だけ」の実践）。

---

## 7条

### 1. 1事実 = 1正典
- ひとつの事実は**1ファイルにだけ**書く。他のファイルからは**参照リンク**を張る。
- コピペで同じ記述を2箇所に置かない。コピーは必ず乖離し、どちらが正しいか分からなくなる。

### 2. 鮮度 > 完全性（特に常時参照する文書）
- **古い記述は「バグ」**。網羅的だが古い文書より、簡潔で正しい文書。
- **仕様を変えたら、同じ変更でドキュメントも直す。** コードとドキュメントは1つの変更単位。
- 「あとで直す」ドキュメントは直されない。その場で直す（→ スキル `engineering-doctrine` 規律1 と同型）。

### 3. 子文書は差分だけ書く
- 共通事項は親（集約ファイル）に1回書く。各サービス/製品の文書は**そことの差分だけ**。
- 共通を各所にベタ書きすると、2条の鮮度が保てなくなる。

### 4. 各文書の冒頭に1行ヘッダ
- すべての文書の先頭に: `> Tier: {{T0/T1/T2}} ・ 最終確認日: {{YYYY-MM-DD}} ・ 1行責務: {{この文書は何のためにあるか}}`
- 「いつ確認した情報か」「正典か参照か」が一目で分かる。鮮度の自己申告。

### 5. 教訓/ノートを足す前に「統合チェック」
- 新しい教訓 memory やノートを書く前に、**既存と束ねられないか**を先に見る。
- 似た教訓が分散すると、どれも参照されなくなる。1テーマ=1ノートに寄せる。

### 6. 葉でなく幹から position する
- 新しい記述を足す時、「これはどの構造・どの章の下に属すか」を先に決める。
- 置き場所を考えずに足すと、文書が「バグリストの寄せ集め」に堕ちる（→ `engineering-doctrine` 規律4 と同型）。

### 7. 常時ロード文書には物理上限がある（サイズ規律 + archive 運用）
- T0（常時ロード）文書には読み込み窓の物理上限がある。超えた分は**黙って**読み飛ばされる — これは「古い記述」（2条）より悪い。書いた本人は載っているつもりで、実際は誰にも読まれていない**静かな嘘**になる。
- 上限を数値で決めて守る（例: index 文書は 24KB 以内・1 entry = 1 行 100-200 字）。超えそうなら**書き足す前に整理**する。
- 完遂済・鮮度切れ（例: 更新から 14 日超の進行記録）は archive ディレクトリへ退避する。**archive ≠ 削除**（git で復元可能・必要なら個別に読みに行く）。index からは外し、archive 索引に 1 行ポインタだけ残す。
- 実害の型: index が上限の 4 倍に肥大 → 後半のエントリが context に載らず、doc と実コードの食い違いが大量発生。整理後に解消。

## Tier 読み分け（いつ何を読むか）

文書には**読む優先度**がある。これを冒頭ヘッダ（4条）で宣言する。

| Tier | 意味 | 例 |
|---|---|---|
| **T0 必読・常時** | 全判断の前提。常にロード | ルート `CLAUDE.md`（北極星 + 指揮系統 + 不変ルール）/ memory |
| **T1 役割別・on-demand** | 特定の作業で読む正典 | 詳細仕様 / 各サービスの CLAUDE.md / 境界マップ |
| **T2 アーカイブ** | 履歴・理論・過去設計 | 設計書群 / 作業ログ / 各文書の歴史節 |

- T0/T1 は **2条（鮮度＞完全性）** を厳格に適用。古い = バグ。
- T2 は「point-in-time の記録」として読む。現在の状態は live を見る（→ `engineering-doctrine` 規律3）。

## 使い方

ドキュメントを作成/更新する時に点検:

```
□ 1. この事実は既にどこかに正典があるか？ あるなら参照リンクにする（コピペしない）
□ 2. 仕様を変えたなら、関連ドキュメントも同じ変更で直したか？ 古い記述を残していないか？
□ 3. 共通事項を子文書にベタ書きしていないか？
□ 4. 冒頭に Tier・最終確認日・1行責務 のヘッダを付けたか？
□ 5. (memory/ノート) 既存と束ねられないか先に見たか？
□ 6. この記述はどの構造の下に属すか、置き場所を決めてから足したか？
□ 7. (常時ロード文書) サイズ上限を超えていないか？ 完遂済・鮮度切れを archive へ退避したか？
```
