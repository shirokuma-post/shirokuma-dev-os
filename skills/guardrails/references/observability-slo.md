# 可観測性 & SLO — 死活監視を「測れる目標」へ

> Tier: T1 ・ 1行責務: health/honeypot 止まりの監視を、構造化ログ・trace・metrics と SLO/エラーバジェットに引き上げ、障害を顧客が気づく前に捕まえる
> 位置づけ: guardrails 層4(実行時の番犬)の強化。死活(up/down)は最低線。ここで「どれくらい健全か(SLI)」「どこまで許すか(SLO)」「いつ動くか(アラート)」を足す。

## 3本柱（最小→充実）
1. **構造化ログ（最初に）**: JSON ログ＋`request_id`/`tenant_id`/`route` を全リクエストに付与。秘密は redact（カタログ A4）。grep でなくフィールド検索できる状態に。
2. **メトリクス**: 4大シグナル＝**遅延・トラフィック・エラー率・飽和**（SRE の Four Golden Signals）。最低 route 別の p50/p95/p99 latency と 5xx 率。
3. **分散トレース**: 余裕が出たら OpenTelemetry で関門↔コンポーネント↔DB を1 trace に。N+1（カタログ G2）や 504連鎖（G1）の原因特定が桁違いに速くなる。

> ベンダー非依存で OpenTelemetry に寄せると、収集先（Datadog/Grafana/Honeycomb 等）を後で替えても計装を捨てずに済む。

## SLI / SLO / エラーバジェット
- **SLI（指標）**: 「良いリクエストの割合」を定義。例: 可用性 = `2xx+3xx+4xx(意図的) / 全リクエスト`、レイテンシ = `p95 < {{500ms}} の割合`。
- **SLO（目標）**: SLI の目標値。例: 可用性 99.9%/30日、レイテンシ SLO p95<500ms を 99%。**100% を目標にしない**（コスト無限＆変更が止まる）。
- **エラーバジェット**: `100% - SLO`。99.9% なら月 ~43分の予算。**予算が残る間は機能開発、燃やし尽くしたら信頼性作業に切替**、という意思決定ルールにする（これが SLO の本質）。
- 定義は `assets/packs/universal/slo.template.yml` に1箇所で（1事実=1正典）。

## アラート（番犬の鳴らし方）
- **症状ベースで鳴らす**（原因でなく顧客影響）。例: 「エラー率 SLO を侵食」「p95 が閾値超」。CPU 高騰だけでは鳴らさない（飽和は補助）。
- **バーンレート・アラート**: エラーバジェットの消費速度で多段階に（速い消費=即ページ / 遅い消費=チケット）。単純閾値より誤報が減る。
- **通知先は責任境界を守る**: 運営最上位の管理者のみ。テナント/系統を越えない（カタログ C2/K4）。dedupe 必須。
- **全アラートに runbook リンク**: 「鳴ったら何を見て何をするか」が無いアラートは無視される。

## health 契約との接続
- 既存の `/health`（healthy/degraded/unhealthy）は**死活の最低線**。SLO は「稼働中の品質」を測る別軸。両方持つ。
- `degraded` は「動くが SLI 劣化」を表現できる。縮退の可視化に使う。
- 死活 cron（smoke）＝即死検知、メトリクス＋SLO＝じわ劣化検知。役割分担。
