# 境界線マップの聞き取り

まず `templates/COMMON-BOUNDARIES.md` の **A（総当たり表）** で、そのサービス固有の境界を生成する。
**そこに載っている 7 問はそのまま使わない**（3 層白ラベル SaaS の例であり、構造が違えば当たらない）。
続いて **B（構造に依らない失敗の型 4 つ）** は、どんなサービスでも必ず当てる。
そのうえで `templates/BOUNDARIES.questions.md` の手順で、載っていない固有の境界を聞き取る。

## この skill が守るべきこと

1. **境界を発明しない。** 聞いて、答えをそのまま記録する。
2. **「分からない」を埋めない。** 可否が未決なら「この線の引き方は事業判断で決まる」と書く。
   実測では、そう書いてあるだけで実装時に AI が空白を見つけて確認しに来た（昇格 3/3）。
   **決めてから書く必要はない。**
3. **誘導しない。** 「普通は見せませんよね」は禁止。判断は事業側のもの。
4. **専門用語を使わない。** RLS / テナント / 認可 と言わず、「見えるか」「変えられるか」で聞く。
5. **区切って聞く。** STEP 1〜2 を確定してから、STEP 3 以降を 3〜5 問ずつ。
6. **書き起こしたら必ず見せて、間違いを指摘してもらう。**

## なぜ聞き取りなのか

境界線マップを白紙から「書く」のは難しい。しかし
「〈A〉が〈B〉のデータを全部見られる画面を作りました。困りますか」には答えられる。

本測定で確認されたこと:
- 人は結論を提示されると頷き、**選択肢を提示されると選ぶ**
- 境界マップは**可否を書かなくても機能する**（実装時に AI が止まる）
- 抽象的な理念より、**線を引く相手**を書く方が易しく、効果が高い

## 上限規律

境界は 3〜5 本を目安に。例外が 3 つ以上ぶら下がった境界は、引き方自体を疑う。
肥大化すると読まれなくなる（`MEMORY.md` が 103KB になり後半未読になった前例）。
