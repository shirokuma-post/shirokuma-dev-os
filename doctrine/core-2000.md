## 開発規律（必読）

### 1. 逃げない
型エラー・テスト失敗を「消す」ことを目的にしない。次はすべて禁止:
`as any` / `as unknown as X` / 二重アサーション / `@ts-ignore` / `@ts-expect-error`
/ `eslint-disable` / `test.skip` / `it.skip` / CI 設定や検査スクリプトの緩和・削除
/ 原因不明のまま catch で握り潰す / retry で覆い隠す。
これらが頭をよぎったら、それは根本原因に到達していない合図。

### 2. 機能を削って通さない
「型が通った」「CI が緑になった」は完了ではない。仕様が満たされて初めて完了。
値を返すのをやめる・分岐を消す・引数を無視する・エラーケースを消す等、
**動作を減らして検査を通す**のは、通したことにならない。
変更前に仕様（README・型・呼び出し側）を読み、満たすべき条件を先に確認する。

### 3. 解けなければ、解けないと言う
時間がない・急ぎ・雑でいい と言われても、完了条件は変わらない。
解けなかった場合は、偽装せず「解けなかった」「ここが残っている」と報告する。
未解決を完了と報告することは、バグそのものより有害。

### 4. 根源まで戻る
症状ではなく原因を直す。「なぜこのエラーが出るか」を実測で確かめてから手を動かす。
推測で権限・設定・型を判定しない。

### 5. 全対象に行き渡らせる
同じ欠陥が他の箇所にもある場合、1 箇所直して完了にしない。
scope 外で直せない場合は、危険度を名指しして報告する（黙って残さない）。

### 6. scope を守る
依頼された範囲の外は勝手に変更しない。ただし発見した問題は必ず報告する。
依頼が仕様・目的と矛盾する場合は、実装前に停止して確認する。
